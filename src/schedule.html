9:40, Keynote

Scalable Cluster
From hell to heaven in 30 minutes with MongoDB

by Norberto Leite
Tech Evangelist @ MongoDB

MongoDB is great database, super simple to use and build applications on.

This is an hands-on demonstration of how to set up a scalable, highly available cluster from the very basic instructions and manual work to the usage of MMS and how it simplifies life and workload on our ops teams when running MongoDB.
We will go through the basic steps of manually setting up replica sets and configure, deploy and monitor them. Then we will show simple tricks and helpers to facilitate the integration with other systems like puppet and chef, nagios etc. Then we finalise by showing how MMS takes all that burden and simplifies the deployment into a couple of clicks on a dashboard.

10:15, Keynote

Building a Microservice Architecture on AWS using AWS Lambda and Docker containers

by Danilo Poccia
Technical Evangelist @ Amazon Web Services

The use of microservices as an architectural pattern, decomposing an application into small, independent components, can improve development, deployment and security. We’ll build a real architecture using AWS Lambda to run event-based functions and Amazon EC2 Container Service and AWS Elastic Beanstalk to manage backend and frontend Docker containers. We’ll evolve from a web based interface to a mobile, cross platform architecture, using a least-privilege approach on security based on AWS Identity and Access Management roles.

IaaS beyond Infrastructure: What's cloud after all?

by Flavio Percoco
Engineer @ Red Hat, OpenStack contributor

We've been talking about Infrastructure providers for quite a bit already. OpenStack itself started as such an IaaS and then it quickly became a cloud provide. But what does that mean? What is a cloud provider suppose to provide? What's cloud after all?

Scaling a distributed system goes beyond virtualization, shared storage etc. In order to support on-demand scaling it is necessary to have an easy way to provision and consume the available infrastructure, a way for the services running in it to communicate, etc. Not to mention latency needs, reliability, etc.
This talk aims to answer the above questions and take a step further down the path of explaining what should be considered essential for nowadays needs, especially when those needs require a 'cloud' to be covered.

16:30  #deploy

Room "Einaudi"

Live deployment on stage with Docker
and NScale

by Matteo Collina
Software Engineer @ nearForm

Setting up new deployment environments is hard, error-prone and costly. The first generation of tools alert alert-success - like Chef and Puppet alert alert-success -  made this process feasible by handling how a machine must be configured, but they still require lots of maintenance and tuning.

The new generation of tools are based around immutable containers alert alert-success - mainly Docker-based alert alert-success - and they remove almost completely the need for custom machine configuration. However, these new architectures creates new problems: how do we manage an application composed of 10s or 100s of different containers?  NScale make things easier by providing a declarative environment to specify the system topology, and changing it is just a GIT revision away: nscale will figure out how to migrate from the current system to the desired configuration in a safe way. In this talk we will setup an application composed by multiple services, dockerize it, and deploy it in the Cloud with NScale.

Room "Giolitti"

Building a Distributed Data Ingestion System with RabbitMQ

by Alvaro Videla
Developer Advocate @ Pivotal/RabbitMQ

Your company has servers distributed around the world and you need to process data in a centralised location. The data is produced by applications using different technology stacks and comes from various sources, from web servers to sensors. How could you solve this problem? Enter RabbitMQ.
In this talk we are going to show how to build a system that can ingest data produced at separate geo located areas (think AWS and it's many regions) and replicate it to a central cluster where it can be further processed and analysed. We will present an example of how to build a system like this one by using RabbitMQ Federation to replicate data across AWS Regions and RabbitMQ support for many protocols to produce/consume data.

If you want to learn what else has RabbitMQ to offer beyond simple messaging and queueing, then this is the talk for you.
